# FIX-0002: Normalize URL before Zod validation

**Audit ref:** AUDIT-001, BUG-2
**Assigned to:** Backend
**Priority:** P1 (blocks pipeline)
**Status:** pending

## Problem

The Zod schema for `POST /api/scan/start` uses `.url()` which requires a protocol prefix (`https://`). The `normalizeUrl()` function that adds the protocol runs AFTER validation (line 93), so it never gets a chance to help. Users who type `example.com` (without `https://`) are rejected with "Must be a valid URL." This breaks the spec's "zero friction entry" principle.

## File(s) to modify

- `src/app/api/scan/start/route.ts` — lines 15-34: the `StartScanSchema` Zod object and `normalizeUrl()` function

## Required change

Restructure the Zod schema to normalize the URL first, then validate. The `normalizeUrl()` function already exists (lines 39-51) and correctly adds `https://` when missing. Move the normalization into the Zod chain using `.transform()`.

**Before:**
```typescript
const StartScanSchema = z.object({
  url: z
    .string()
    .min(1, 'URL is required')
    .url('Must be a valid URL')
    .refine(
      (val) => {
        try {
          const parsed = new URL(val);
          return ['http:', 'https:'].includes(parsed.protocol);
        } catch {
          return false;
        }
      },
      { message: 'URL must use http or https protocol' }
    ),
  utmSource: z.string().optional(),
  utmMedium: z.string().optional(),
  utmCampaign: z.string().optional(),
});
```

**After:**
```typescript
const StartScanSchema = z.object({
  url: z
    .string()
    .min(1, 'URL is required')
    .transform((val) => {
      const trimmed = val.trim();
      if (!/^https?:\/\//i.test(trimmed)) {
        return `https://${trimmed}`;
      }
      return trimmed;
    })
    .pipe(
      z.string().url('Must be a valid URL').refine(
        (val) => {
          try {
            const parsed = new URL(val);
            return ['http:', 'https:'].includes(parsed.protocol);
          } catch {
            return false;
          }
        },
        { message: 'URL must use http or https protocol' }
      )
    ),
  utmSource: z.string().optional(),
  utmMedium: z.string().optional(),
  utmCampaign: z.string().optional(),
});
```

Also update the POST handler to use the already-normalized URL from Zod output instead of calling `normalizeUrl()` separately. On line 93, `const normalizedUrl = normalizeUrl(url);` can become `const normalizedUrl = url;` since the URL is already normalized by the schema's `.transform()`. Alternatively, keep calling `normalizeUrl(url)` — it's idempotent and won't hurt.

## Constraints

- Do NOT change the `normalizeUrl()` function itself — it may be used elsewhere
- Do NOT modify any other route or file
- The validation must still reject non-http/https protocols (e.g., `javascript:`, `ftp:`)
- The schema must accept both `example.com` and `https://example.com`

## Verification

The Orchestrator will verify by:
1. `npx tsc --noEmit` — zero errors
2. Confirm the Zod schema uses `.transform()` before `.url()`
3. Mentally trace: input `"example.com"` → transform adds `https://` → `.url()` passes → `.refine()` passes → output is `"https://example.com"`
