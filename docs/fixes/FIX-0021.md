# FIX-0021: Social disambiguation — emit social_ambiguous SSE event

**Audit ref:** AUDIT-003, Phase A item 2
**Assigned to:** Backend
**Priority:** P3 (reliability)
**Status:** pending

## Problem
The `social_ambiguous` SSE event is defined in `contracts/events.ts:45-49` and handled by the frontend (`ScanLayout.tsx:183`), but it is **never emitted** anywhere in the backend. When `social-detector.ts` finds multiple handles for the same platform, it silently picks one (nav-area heuristic or first match) instead of surfacing the ambiguity to the user. The SocialConfirmation modal never appears.

## File(s) to modify
- `src/lib/screenshots/social-detector.ts` — around line 175-205: when `uniqueMatches.length > 1`, return ambiguous options instead of silently resolving
- `src/lib/screenshots/pipeline.ts` — where social detection results are processed: check for ambiguous results and store them for SSE emission
- `src/app/api/scan/status/[id]/route.ts` — around line 159-174: emit `social_ambiguous` event when ambiguous socials are detected (in addition to `social_detected` for resolved ones)

## Required change
1. **social-detector.ts:** When multiple unique handles are found for a single platform and there's no clear winner (e.g., no single handle appears in nav area), return them as an ambiguous set. Add an `ambiguous` field to the return type or return a separate structure.
2. **pipeline.ts:** After running social detection, check for ambiguous results. Store them in the scan record (e.g., `detected_socials_ambiguous` or a JSONB field) so the SSE status route can pick them up.
3. **status route:** When polling scan state, check for unresolved ambiguous socials. Emit `social_ambiguous` event with `{ platform, options: [{ handle, url }] }` matching the contract type. Only emit once per platform (track emission state).
4. When the user resolves ambiguity via `POST /api/scan/capture-info`, mark the platform as resolved so the event stops being emitted.

## Constraints
- Do NOT modify Frontend files — the SocialConfirmation component already handles this event correctly
- Do NOT modify `contracts/events.ts` — the event type is already correct
- The fix must conform to `ScanSSEEvent` union type in `contracts/events.ts`
- Maintain backward compatibility: scans with clear single handles should still work exactly as before

## Verification
The Orchestrator will verify by:
1. Grepping for `social_ambiguous` in the backend — confirming it is now emitted from the status route
2. Reading social-detector.ts to confirm ambiguous detection logic exists
3. Confirming the emitted event shape matches `contracts/events.ts:45-49`
4. `npx tsc --noEmit` passes
