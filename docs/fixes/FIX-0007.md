# FIX-0007: URL input type="url" blocks bare domains via browser validation

**Audit ref:** AUDIT-002, E2E browser test
**Assigned to:** Frontend
**Priority:** P1 (blocks pipeline — users can't start a scan with bare domains)
**Status:** pending

## Problem

The HeroSection URL input uses `type="url"` (line 114), which triggers the browser's native HTML5 validation. Browsers require a protocol prefix for `type="url"` inputs — entering `example.com` or `www.example.com` shows a browser tooltip like "Please enter a URL" and prevents form submission entirely. The Backend already normalizes bare domains (FIX-0002), but the request never reaches the API because the browser blocks it.

## File(s) to modify

- `src/components/landing/HeroSection.tsx` — line 114: change `type="url"` to `type="text"`

## Required change

Change the input type from `url` to `text`. The placeholder already hints at the expected format. The Backend handles normalization.

**Before:**
```tsx
<input
  type="url"
  value={url}
  onChange={(e) => setUrl(e.target.value)}
  placeholder="https://yourwebsite.com"
  required
```

**After:**
```tsx
<input
  type="text"
  value={url}
  onChange={(e) => setUrl(e.target.value)}
  placeholder="yourwebsite.com"
  required
```

Also update the placeholder from `https://yourwebsite.com` to `yourwebsite.com` — since we now accept bare domains, the placeholder should reflect that (zero friction).

## Constraints

- Do NOT modify the form's `onSubmit` handler or any other logic
- Do NOT add client-side URL validation — the Backend handles it
- Only change the `type` attribute and `placeholder` text

## Verification

The Orchestrator will verify by:
1. `npx tsc --noEmit` — zero errors
2. Confirm `type="text"` on the input
3. Confirm placeholder is `yourwebsite.com` (no protocol prefix)
